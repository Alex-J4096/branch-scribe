# BranchScribe 记忆功能说明书

本文说明 BranchScribe 中角色卡、地点卡、世界规则、Memory、角色状态、伏笔和时间线的用途、工作原理与使用方法。

## 1. 先理解三类“记忆”

BranchScribe 没有把所有资料都塞进同一个记忆库，而是按用途分成三层：

| 层级 | 功能 | 适合保存 | 是否自动进入普通 LLM 写作上下文 |
| --- | --- | --- | --- |
| 硬设定（Canon） | 角色卡、地点卡、规则 | 不应轻易违背的事实 | 角色、地点需先关联当前 Block；`canon` 状态的规则全局自动加入 |
| 语义记忆（Memory） | Memory Chunk | 经历、对话、伏笔片段、背景资料等可检索内容 | 系统按当前内容关键词检索，最多加入 5 条 |
| 剧情工程台账 | 角色状态、伏笔、时间线 | 状态变化、伏笔生命周期、事件顺序 | 当前不会自动加入普通写作上下文 |

简单判断：

- “模型必须遵守”——写入 Canon。
- “需要时让系统想起来”——写入 Memory。
- “我需要追踪和管理”——写入角色状态、伏笔或时间线。

## 2. 入口与推荐工作流

进入项目工作台后，顶部可打开“角色”“地点”“规则”“Memory”“伏笔”“时间线”。

推荐顺序：

1. 先建立常用角色卡、地点卡和世界规则。
2. 编辑 Block 时，在 Block 详情的 Metadata 区选择本段出现的角色和地点并保存。
3. 把重要经历、对话、背景信息或需要被召回的线索整理为 Memory。
4. 用伏笔面板跟踪埋设与回收，用时间线整理事件先后。
5. 生成前打开上下文预览，确认实际将发送给模型的 Canon、Memory 和前文。

上下文预览是最终依据：资料已保存，不代表本轮一定会被模型看到。

## 3. 角色卡

### 3.1 用途

角色卡属于 Canon 硬设定，适合记录：

- 姓名、别名和称呼；
- 外貌、身份、性格、能力、关系；
- 不应被普通续写随意改变的事实。

表单字段：

- `名称`：必填，也是检索和展示的主名称。
- `别名`：用逗号分隔，可填写昵称、代号、旧名。
- `描述`：面向作者和模型的自然语言说明。
- `状态`：`canon`、`draft` 或 `deprecated`。
- `重要度`：1–10；多个设定进入上下文时，重要度高的排在前面。
- `Attributes JSON`：结构化扩展字段，例如 `{"年龄": 24, "阵营": "巡夜司"}`。

### 3.2 如何让角色卡影响写作

仅创建角色卡还不够。打开某个 Block，在详情的 Metadata 区勾选“出现角色”并保存。之后该 Block 的续写、改写、一致性检查和事件提取才能读取这些角色卡。

当前筛选规则：

- 与当前 Block 关联；
- 状态不是 `deprecated`；
- `draft` 角色也会被读取，因此草稿设定若不希望影响生成，应取消 Block 关联或标记为 `deprecated`。

### 3.3 从后续剧情更新角色卡

角色列表中的星光按钮可从正文提取新版角色卡：

1. 选择一个起始 Block。
2. 系统列出同一分支中从该节点开始的后续 Block，并默认全选。
3. 取消不需要参与汇总的 Block，选择 LLM 模型。
4. 点击“生成候选角色卡”。
5. 检查并编辑完整描述、Attributes 和变化摘要。
6. 点击“保存为新版本”。

保存后会同时：

- 更新当前角色卡；
- 在角色状态中追加一份 `character_card` 完整快照；
- 保存来源 Block、模型、Generation Run 和变化摘要。

旧快照不会被覆盖，可在角色卡下方的“历史版本”查看。模型生成的只是候选，保存前应人工核对。

## 4. 地点卡

地点卡也是 Canon，适合记录空间布局、气候、交通、出入口、危险、文化习俗和不能随剧情任意改变的环境事实。

创建和编辑字段与角色卡相同。要让地点卡进入某个 Block 的生成上下文，需要在 Block Metadata 的“地点”中选择它并保存。每个 Block 当前只能选择一个地点。

地点卡状态规则与角色卡一致：`deprecated` 不进入上下文，`canon` 和 `draft` 在被关联后都可进入。

## 5. 世界规则

规则用于保存对整个项目都成立的硬约束，例如：

- 魔法使用代价；
- 科技水平和物理限制；
- 社会制度、禁忌和称谓规范；
- 叙事中不可破坏的世界运行规律。

与角色、地点不同，状态为 `canon` 的规则不需要关联 Block，会自动加入每个 Block 的 Canon 上下文。`draft` 和 `deprecated` 规则不会自动加入。

因此：

- 已确定并需要模型遵守的规则设为 `canon`；
- 尚未定稿的规则设为 `draft`；
- 已废止但仍需留档的规则设为 `deprecated`。

规则应尽量短、明确、可验证。与其写“魔法很危险”，不如写“每次施法都会永久损失一段与施法目标相关的记忆”。

## 6. Memory

### 6.1 用途与工作原理

Memory Chunk 是可检索的语义资料，不是强制约束。适合保存：

- 某角色过去的经历或重要对话；
- 一段地点描写；
- 章节之外的背景资料；
- 可能在后文被再次提及的线索；
- 正文片段或人工整理的简短事实。

普通生成时，系统从用户指令、选中文本、当前正文以及已关联 Canon 的名称和别名中提取关键词，在当前项目的 Memory 文本和标签中匹配，按新到旧最多加入 5 条。没有匹配关键词时不会加入 Memory。

注意：Memory 管理页提供向量语义搜索，但当前普通写作的自动上下文召回使用关键词匹配，并不直接使用向量相似度排序。

### 6.2 手动创建

主要字段：

- `Source Type`：来源类型，手工资料通常填 `manual`。
- `Source ID`：可选的来源 ID。
- `Chunk Text`：真正参与搜索和上下文注入的正文。
- `Kind`：资料类别，如 `note`、`dialogue`、`background`。
- `Tags`：逗号分隔；标签也参与自动关键词匹配。
- `Metadata`：扩展 JSON，仅用于记录额外信息。

一条 Memory 最好只表达一个相对完整的事实，并在正文或标签中写出角色名、地点名等可被匹配的关键词。

### 6.3 从 Block 创建

在 Memory 页面选择一个 Block，可将它的当前 Revision 转成 Memory。系统会：

- 读取该 Block 当前版本；
- 将 HTML 转为纯文本；
- 记录来源 Revision；
- 创建 `block_revision` 类型的 Memory。

后续修改原 Block 不会自动同步这条 Memory；需要删除、编辑或重新创建。

### 6.4 关键词搜索、语义搜索与 Reindex

- 关键词搜索：直接匹配 Memory 文本，可叠加 Kind 和 Tag 筛选。
- 语义搜索：需要全局 Embedding Profile，按含义而非字面词搜索。
- Reindex：用所选 Embedding Profile 重新生成当前项目全部 Memory 和 Canon 的向量索引。

编辑 Memory 文本或 Canon 的名称、别名、描述、Attributes 后，对应旧向量会失效；再次使用语义搜索前应执行 Reindex。Reindex 不影响普通写作上下文中的关键词召回。

## 7. 角色状态

角色状态用于保存角色在剧情某一时点的状态快照，例如伤势、位置、持有物、关系变化或阶段性角色卡。

每条状态可关联：

- 角色；
- 来源 Block；
- 状态键和值；
- 发生时间和备注；
- 来源模型、Generation Run 等 Metadata。

当前界面主要通过“从后续剧情提取新版角色卡”生成并展示 `character_card` 状态历史。它的价值是版本追溯，不会自动把所有历史状态加入普通写作上下文。需要模型强制遵守的最新状态，应确认已经同步到当前角色卡；需要按语义召回的经历，可另存为 Memory。

## 8. 伏笔

伏笔面板用于管理线索生命周期，而不是自动生成内容。

状态含义：

- `已埋设`：线索已出现，尚未展开。
- `发展中`：线索正在被强化或推进。
- `已回收`：谜底或作用已经兑现。
- `已废弃`：明确不再使用。

使用方法：

1. 新建伏笔，填写一句话标题和详细说明。
2. 关联“埋设 Block”。
3. 剧情推进时将状态改为“发展中”。
4. 回收后改为“已回收”，并关联“回收 Block”。
5. 用状态筛选检查长期未处理的伏笔。

伏笔记录当前不会自动进入普通生成上下文。如果希望模型在续写时主动想起某条伏笔，可将关键内容另存为带角色名、地点名或线索关键词的 Memory，并在生成前检查上下文预览。

## 9. 时间线

时间线用于整理故事内事件顺序。每条事件包含：

- 标题和描述；
- 故事内时间表达，如“第三日午夜”；
- 数字排序值；
- 来源 Block；
- 可选的相关 Canon。

列表按 `排序` 值排列；`故事内时间`主要用于显示，不会自动解析成日期或决定排序。

### 9.1 手动维护

新建事件后填写时间、排序、来源 Block 和相关 Canon。若两个事件的时间表达无法直接比较，使用排序值明确先后。

### 9.2 从 Block 提取

1. 在“从 Block 提取事件”中选择正文和 LLM 模型。
2. 点击“提取并加入时间线”。
3. 系统只要求模型提取正文中明确发生的事件，不提取计划、假设或仅被提及的往事。
4. 提取结果会直接创建为时间线事件，并记录模型和 Generation Run。
5. 完成后人工编辑时间表达、排序与关联信息。

提取时模型只能关联当前 Block 可见的 Canon；无效关联会被系统移除。时间线当前不会自动进入普通写作上下文。

## 10. 这些资料如何进入一次 LLM 生成

构建普通写作上下文时，系统按以下顺序组织资料：

1. 当前 Block；
2. 当前 Block 关联的角色、地点，以及全局 `canon` 规则；
3. 选中文本；
4. 分支或章节摘要；
5. 当前故事路径上的前文 Block；
6. 关键词匹配到的 Memory。

系统根据模型上下文窗口、最大输出长度和预留空间计算预算。当前 Block 和 Canon 属于必选项；其他项目可能因用户临时取消或 token 预算不足而不进入最终 Prompt。

用户可在 LLM Chatbox 的上下文设置中选择前文节点数量，并在上下文预览中：

- 查看每条来源及预计 token；
- 临时排除非必选项；
- 查看 System Prompt、User Prompt 和最终 Prompt。

生成记录会保存本轮上下文快照，便于之后追查模型当时实际看到了什么。

## 11. 一致性检查

Block 详情中的一致性检查会读取：

- 当前 Block 正文；
- 本 Block 关联的角色与地点；
- 全局 `canon` 规则。

模型只应报告有明确正文证据的冲突，并给出关联 Canon、严重度、原因和修订建议。检查结果是辅助判断，不会自动修改正文或设定。

如果某条设定没有关联当前 Block（且不是全局规则），它不会参与本次检查。

## 12. 常见误区与排查

### “我创建了角色卡，模型为什么没用？”

检查角色是否已在当前 Block Metadata 中勾选并保存，再查看上下文预览。

### “规则为什么没有进入上下文？”

只有状态为 `canon` 的规则会全局自动加入；检查是否误设为 `draft` 或 `deprecated`。

### “Memory 页面语义搜索能找到，生成时却没召回？”

管理页语义搜索使用向量；普通生成当前使用关键词匹配。给 Memory 增加明确名称和标签，或在本轮指令中提及对应关键词。

### “更新了 Block，旧 Memory 会同步吗？”

不会。从 Block 创建的 Memory 是当时当前 Revision 的快照，需要手工更新或重新创建。

### “时间线、伏笔已经保存，模型会自动知道吗？”

当前不会。它们是管理台账。需要参与生成时，可把关键内容转成 Memory，或直接写入本轮指令。

### “角色历史版本会自动恢复吗？”

不会。历史版本用于追溯；当前写作读取的是角色卡当前内容。要回退需根据历史快照手工更新当前角色卡。

## 13. 最佳实践速查

- Canon 写稳定事实，Memory 写可召回资料，台账写过程状态。
- 每个 Block 及时关联出现角色和地点。
- 全局规则保持短、硬、可验证。
- Memory 一条一事，并添加能在正文中出现的标签。
- LLM 提取结果先人工核对，再作为正式资料保存。
- 每次关键生成前看上下文预览，不凭感觉判断模型“应该知道”。
- 角色状态变化后同步更新当前角色卡；重要经历另存 Memory。
- 定期检查未回收伏笔和时间线排序。
